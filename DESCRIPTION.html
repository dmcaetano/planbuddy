PlanBuddy

One confident, lockable plan for the people involved — Day off, Weekend, Getaway, or Vacation.

React + Vite + TS Express + TS Neon / PGlite DeepSeek V4 Flash + demo AI

PlanBuddy turns "what should we do?" into one lockable plan. This is the local v1 MVP built to the frozen product contract: a single-package React client and Express API, normalized Postgres with a zero-setup local fallback, secure auth, full-CRUD household memory, four planning modes, and a deterministic recommendation engine sitting on top of a pluggable AI proposer.

The smallest lovable loop

  1. Pick a scale, the people (and pets) involved, and a date window; add an optional context note.
  2. Press Plan it.
  3. Get one confident recommendation with an honest, memory-grounded rationale.
  4. Lock it, browse a diverse alternate, say Not this with a reason, or tweak the spec.
  5. Rate it afterward and optionally comment.
  6. Anything learned shows up in Memory with full provenance and full CRUD.

Stack

Client
React + Vite + TypeScript, mobile-first (≤640px), no raster assets — warm "daylight editorial" design system + lucide-react vector icons.
Server
Express + TypeScript, serves the built client and a JSON API under /api.
Database
Normalized Postgres, Neon-ready. No DATABASE_URL? Falls back to an embedded PGlite database — same schema, zero setup.
Auth
Argon2id (bcrypt cost-12 fallback), 256-bit opaque session tokens in a signed httpOnly SameSite=Lax cookie.
AI
DeepSeek V4 Flash via OpenRouter when a key is set; otherwise a deterministic local demo AI — fully usable offline.
Weather / Places
Open-Meteo (no key). Pluggable place resolver; defaults to explicit Inspiration mode.

Deterministic recommendation pipeline

Per the contract's core doctrine — DeepSeek proposes, server code disposes — every AI-proposed candidate passes a deterministic, replayable pipeline before it's shown:

  • Constraint filter — mechanical keyword/structure check, independent of the model's own self-report. Rejects constraint violations, invalid citations, duplicates, impossible travel radius, and venue-firewall violations.
  • Scoring — base fit 0.5 per participant; loves/avoids shift it; hunches contribute at most ±0.15 and never appear in rationales; group fit is the minimum per-participant fit (least misery); feasibility folds in weather/travel; novelty penalizes recent repeats. Final score = 55% group fit + 25% feasibility + 20% novelty.
  • Alternates — drawn from the same filtered batch, category-diverse from the winner.

Memory model

  • Constraints — hard vetoes. Typed constraints are verified immediately. Chat-extracted constraints become active_unverified (and start protecting immediately) only after the server verifies the model's quote is a verbatim substring of the source message at the claimed offsets — quote or demote. A failed quote becomes a non-filtering hunch instead.
  • Tastes — loves/avoids that shape scoring but never veto.
  • Hunches — weak, decaying, non-citable. Promote to a taste (never a constraint) after three independent evidence events from distinct plans/sessions; decay after six unreinforced generations or 90 days.

Running it

npm install
npm run dev      # Vite (5173) + Express (4000), proxied
# or:
npm run build
npm start          # single process on PORT

No .env is required — copy .env.example for a real Neon DB or OpenRouter key.

Verifying it

npm run typecheck   # strict TS, client + server
npm run lint          # ESLint
npm run build          # vite build + tsc build
npm test                 # Vitest: unit + AI-contract + Supertest integration
npm run test:e2e        # Playwright: signup → onboard → generate → reject → lock → feedback → Memory

All green as of this build: 59 Vitest tests, 1 Playwright scenario, zero TypeScript/ESLint errors.

Out of scope for v1

  • No live place-resolver provider wired up (interface is pluggable; Inspiration mode by default).
  • No Render/Neon deployment performed — local, build-verified work on build-mvp, not pushed.
  • No booking or detailed itinerary generation, by design.
